## Data loading and memory usage 
მონაცემები, რომლებიც გაყოფილი იყო ორ ბაზად გავაერთიანე left join-ის დახმარებით.

**Memory Optimization**: შევქმენი ფუნქცია reduce_mem_usage, რომელმაც მონაცემთა ტიპების ოპტიმიზაციის გზით (მაგ. float64 -> float32) მეხსიერების მოხმარება მნიშვნელოვნად შეამცირა, რაც კრიტიკული იყო მასშტაბური გამოთვლებისთვის.

# Feature Engineering
მანამ სანამ დავიწყებდი მიდგომების შემუშავებას, დავაკვირთი მონაცემებს და დავიწყე მისი თავისებურებების შესწავლა და ანალიზი. 

### Missing Values
იმისათვის, რომ დამენახა რამდენად კრიტიკული იყო NaN value-ების რაოდენობა feature-ებში, შევხედე მათ ვიზუალურად. მაგალითად, ჯერ დავაკვირდი Identity Column-ებში missing value-ების განაწილებას.

<img width="988" height="552" alt="Screenshot 2026-05-04 233423" src="https://github.com/user-attachments/assets/71e3813b-08ea-4ec5-bf54-5d5b30f00c57" />

გრაფიკიდან ჩანს, რომ Identity ცხრილის მონაცემებში (სვეტები id_01-დან id_38-მდე) გამოტოვებული მნიშვნელობების წილი კრიტიკულად მაღალია. ბევრ სვეტში NaN-ების რაოდენობა აჭარბებს 90%-ს.დასკვნა. ეს ნიშნავს, რომ ეს სვეტები თითქმის არ შეიცავს ინფორმაციას ტრანზაქციების უმეტესობისთვის.

სწორედ ამიტომ შევქმენენი DropMissingFeatures კლასი. თუ ჩვენს DropMissingFeatures კლასში threshold პარამეტრს დავაყენებთ მაგალითად 0.9-ზე, ყველა ის სვეტი, რომელიც წითელ წყვეტილ ხაზს ზემოთ მოექცა, ავტომატურად წაიშლება Pipeline-ის მიერ. ეს ასევე დაგვეხმარება მოდელის გადატვირთვის (Overfitting) თავიდან აცილებაში, რადგან მოდელი აღარ შეეცდება "ისწავლოს" იმ მცირე მონაცემებიდან, რომლებიც ამ ცარიელ სვეტებშია დარჩენილი.


<img width="836" height="432" alt="Screenshot 2026-05-04 233430" src="https://github.com/user-attachments/assets/1cd284cc-c77d-4fdc-8c40-e3ec4a7e94f6" />

Identity მონაცემების გარდა, გამოტოვებული მნიშვნელობების მაღალი წილი შეინიშნება ტრანზაქციების ცხრილის V1-დან V339-მდე სვეტებშიც. ჩემი სტრატეგია ამ შემთხვევაშიც DropMissingFeatures კლასის გამოყენება იყო. წავშალე ის სვეტები, სადაც ინფორმაციის ნაკლებობა მოდელს ხელს შეუშლიდა (Threshold 0.9).

**NaN Value-ების შევსება (Imputation):** იმ სვეტებში, რომლებიც გრაფიკზე ჩანს (28% - 45% NaN), მონაცემები არ წავშალე, რადგან ისინი მაინც შეიცავენ მნიშვნელოვან ინფორმაციას. მათ შესავსებად გამოვიყენე მედიანური მნიშვნელობები.

## კატეგორიული ცვლადების რიცხვითებში გადაყვანა (Categorical Encoding)
ვინაიდან მონაცემები შეიცავს ბევრ კატეგორიულ სვეტს (მაგალითად: ბარათის ტიპი, დომენი, მოწყობილობის მოდელი), შეიქმნა სპეციალური კლასი CategoricalEncoder, რომელიც უზრუნველყოფს მათ ტრანსფორმაციას რიცხვით ფორმატში.
1. Ordinal Encoding: გამოყენებულ იქნა OrdinalEncoder, რომელიც თითოეულ უნიკალურ კატეგორიას ანიჭებს შესაბამის მთელ რიცხვს. ეს მიდგომა განსაკუთრებით ეფექტურია ხეზე  დაფუძნებული მოდელებისთვის (XGBoost, LightGBM), რადგან ის არ ზრდის მონაცემთა განზომილებას (განსხვავებით One-Hot Encoding-ისგან).

2. Unknown Values Handling: კოდში გათვალისწინებულია handle_unknown='use_encoded_value' პარამეტრი. ეს ნიშნავს, რომ თუ მოდელს ტესტირების (Inference) დროს შეხვდება ისეთი კატეგორია, რომელიც წვრთნისას არ უნახავს, ის მას ავტომატურად მიანიჭებს სპეციალურ მნიშვნელობას, ნაცვლად იმისა, რომ შეცდომა დააბრუნოს.

### ახალი Feature-ების შემოღება
#### დროის Feature-ები
მონაცემებში მოცემული იყო TransactionDT სვეტი, რომელიც წარმოადგენს წამებს რაღაც საწყისი წერტილიდან. ჩემი იდეა, გამომეყო საათის (Hour) სვეტი, ეყრდნობოდა შემდეგ ლოგიკას:რეალურ სამყაროში თაღლითური ქმედებები ხშირად ღამის საათებში აქტიურდება, როცა ადამიანებს სძინავთ და ბარათზე უცხო აქტივობის შემჩნევის ალბათობა ნაკლებია.
და ლეგიტიმური მომხმარებლები, როგორც წესი, ყიდულობენ დღის გარკვეულ მონაკვეთში. თუ ტრანზაქცია ხდება ღამის 4 საათზე, მოდელმა ეს ინფორმაცია უნდა გამოიყენოს, როგორც დამატებითი "ეჭვის" საფუძველი.

სწორედ ამიტომ დავაკვირდი დროს საათების მიხედვით:

ღამის პერიოდი (საათი 0-დან 10-მდე): წითელი ფართობი (Fraud) უფრო დიდია და უფრო "გაბერილი", ვიდრე ლურჯი (Legit). ეს ნიშნავს, რომ მიუხედავად იმისა, რომ ღამით ტრანზაქციების საერთო რაოდენობა მცირდება, თაღლითობის კონცენტრაცია ამ დროს გაცილებით მაღალია.

დღის პერიოდი (საათი 12-დან 22-მდე): აქ ლურჯი ხაზი (Legit) უფრო მაღლაა. ეს ლოგიკურია – ადამიანები დღისით უფრო აქტიურად ყიდულობენ და თაღლითური ტრანზაქციები ამ დიდ ნაკადში "იკარგება".
გადაკვეთის წერტილები: დაახლოებით 11 საათზე და 23 საათზე ხდება პატერნის ცვლილება.

თუ მოდელს მხოლოდ TransactionDT-ს (წამებს) მივცემდით, ის ამას კარგად ვეღარ დააკვირდებოდა. ახლა კი, როცა შენი TimeFeaturesEncoder კლასი შექმნის hour სვეტს, მოდელი მარტივად ისწავლის კანონზომიერებას.
ამ ინფორმაციის გამოსაყენებლად, Pipeline-ში ვამატებ TimeFeaturesEncoder-ს, რომელიც TransactionDT-დან ამოიღებს საათობრივ მონაცემებს.

კვირის დღეების მიხედვით ტრანზაქციების რაოდენობა იცვლება, რაც მიუთითებს იმაზე, რომ მომხმარებელთა ქცევა კვირის განმავლობაში არაერთგვაროვანია. მიუხედავად იმისა, რომ თაღლითობის აბსოლუტური რაოდენობა სტაბილური ჩანს, მისი წილი საერთო ტრანზაქციებში იზრდება იმ დღეებში, როცა აქტივობა დაბალია (მაგ. დღე 3). ეს კიდევ ერთხელ ასაბუთებს TimeFeatures ენკოდერის საჭიროებას.

#### aggregation features
მოდელის სიზუსტის გასაზრდელად გამოვიყენე Group Statistics ტექნიკა. კერძოდ, შევქმენი ახალი ცვლადები, რომლებიც აჩვენებენ მიმდინარე ტრანზაქციის თანხის ფარდობას საშუალო ტრანზაქციასთან card1 და addr1 ჯგუფებისთვის. ამან საშუალება მისცა მოდელს აღმოეჩინა ანომალიური ხარჯვა, რაც თაღლითობის ერთ-ერთი ძირითადი ინდიკატორია.
